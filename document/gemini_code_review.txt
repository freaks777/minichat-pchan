### 🚨 1. 致命的なバグ：`escapeHtml` が全く機能していない（XSS脆弱性）
**対象ファイル:** `sessions.js`, `session-setup.js`, `settings.js`

この3つのファイルの末尾に定義されている `escapeHtml` 関数に**致命的なタイポ**があります。

**【現在の危険なコード】**
```javascript
function escapeHtml(str) {
  // ❌ 置換先がエスケープ文字になっておらず、元の文字をそのまま返している！
  return str.replace(/&/g, '&').replace(/</g, '<').replace(/>/g, '>').replace(/"/g, '"');
}
```
この状態だと、`<` は `<` のまま出力されてしまいます。
ペルソナ名や、StudioでLLMが自動抽出した値（設定のSubject等）に `<script>` などのタグが含まれていた場合、**`innerHTML` で描画された瞬間にアプリ内で悪意あるプログラムが実行されてしまいます（Stored XSS）。**

**【修正コード】**
以下のように、`studio.js` で書かれているものと同じ、正しいHTMLエンティティ（`&amp;` や `&lt;` など）に置換するように修正してください。さらに、シングルクォート（`'`）もエスケープしておくと完璧です。

```javascript
function escapeHtml(str) {
  if (!str) return "";
  return String(str)
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;')
    .replace(/'/g, '&#39;');
}
```
※上記コードを `sessions.js`, `session-setup.js`, `settings.js` の3ファイルの `escapeHtml` に上書きしてください。

### ⚠️ 2. `chat.js` の `escapeHtml` のエスケープ漏れ
**対象ファイル:** `chat.js`

`chat.js` の `escapeHtml` は `&lt;` などに正しく変換されていますが、**ダブルクォート（`"`）とシングルクォート（`'`）のエスケープが漏れています。**

**【現在のコード】**
```javascript
function escapeHtml(str) {
  return str.replace(/&/g, "&amp;").replace(/</g, "&lt;").replace(/>/g, "&gt;");
}
```

現状の実装（`div.innerHTML = ... <div>${escaped}</div>`）であれば、タグの内側（テキストノード）に入っているだけなので直ちにXSSにはなりません。
しかし、将来的に `<div title="${escaped}">` のように**HTMLの属性値**にこの関数を使った瞬間、ダブルクォートを閉じられてスクリプトを注入される脆弱性になります。

**【修正コード】**
こちらも先ほどと同じ、完全なエスケープ関数に置き換えておくことを強く推奨します。
```javascript
function escapeHtml(str) {
  if (!str) return "";
  return String(str)
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;')
    .replace(/'/g, '&#39;');
}
```

### ⚠️ 1. データロストのリスク：`history.py` のファイル上書き
**対象ファイル:** `core/history.py` （`_save_full` メソッドなど）

`_save_full` では、履歴を以下のように `"w"` モードで直接上書き保存しています。
```python
    def _save_full(self):
        self.persona_dir.mkdir(parents=True, exist_ok=True)
        lines = [json.dumps(m, ensure_ascii=False) for m in self._messages]
        # ❌ 直接上書きモードで開いている
        with open(self.today_file, "w", encoding="utf-8") as f:
            f.write("\n".join(lines) + "\n")
```

**【なぜ危険なのか？】**
Python（およびOS）の仕様上、`open(..., "w")` を実行した瞬間に、**そのファイルは一旦0バイトに切り詰められます**。
もしこの書き込み処理の最中に、「アプリがクラッシュした」「PCがフリーズして強制終了した」「停電した」などの不測の事態が起きると、書き込み途中のデータも元のデータもすべて失われ、**ファイルが0バイトになって履歴が完全消滅**します。

**【修正コード（アトミックな書き込み）】**
データを守るための鉄則として、**「一時ファイルに書き込んでから、一瞬でリネーム（置換）する」**方法に変更してください。これにより、書き込み中に落ちても古いファイルが必ず残ります。

```python
    def _save_full(self):
        """全メッセージをJSONLファイルに書き出す（アトミック上書き）。"""
        self.persona_dir.mkdir(parents=True, exist_ok=True)
        lines = [json.dumps(m, ensure_ascii=False) for m in self._messages]
        
        # 🔽 一時ファイルに書き込む
        temp_file = self.today_file.with_suffix(".jsonl.tmp")
        with open(temp_file, "w", encoding="utf-8") as f:
            f.write("\n".join(lines) + "\n")
            
        # 🔽 書き込み成功後、元のファイルと一瞬で入れ替える（アトミック操作）
        import os
        os.replace(temp_file, self.today_file)
```

### 💡 2. 隠れたバグの種：`validate_persona_id` の正規表現
**対象ファイル:** `core/persona_manager.py`

パストラバーサルを防ぐために書かれている正規表現に、Python特有の罠があります。

```python
# 現在のコード
_PERSONA_ID_RE = re.compile(r"[\w\-]+")
```

**【なぜ問題なのか？】**
Python 3 では、**`\w` は半角英数字だけでなく、日本語（ひらがな、漢字、全角文字）にもマッチ**してしまいます。
フロントエンド（UI側）では `ペルソナID（半角英数）` と指定しているにもかかわらず、バックエンド側では `ペルソナあいうえお` のような全角のディレクトリ作成を許容してしまいます。
全角文字のディレクトリ名が混じると、Windows環境などでのパス解決やZIP化バックアップ時などに、予期せぬ文字化けやファイルアクセスエラーを引き起こす原因になります。

**【修正コード】**
純粋な半角英数字とハイフン、アンダースコアだけに限定するために、以下のように明示的に書くか、`re.ASCII` フラグを付けてください。

```python
# 修正案1：明示的に文字クラスを指定する（おすすめ）
_PERSONA_ID_RE = re.compile(r"^[a-zA-Z0-9_\-]+$")

# または 修正案2：re.ASCII フラグをつける
_PERSONA_ID_RE = re.compile(r"^[\w\-]+$", re.ASCII)
```
※先頭 `^` と末尾 `$` をつけることで、部分一致によるすり抜けも確実に防げます（現在は `fullmatch` を使っているので大丈夫ですが、正規表現自体に書いておくとより安全です）。
